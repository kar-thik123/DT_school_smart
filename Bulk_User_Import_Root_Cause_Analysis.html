RCA: Browser vs. Runtime Execution Mismatch

Deep dive analysis of why the browser requests fail while direct runtime scripts succeed.

Backend Runtime: SUCCESS (200 OK) Browser Execution: FAILS (500 Error)

RCA Diagnostic Summary

A comprehensive comparison was executed between a simulated browser-style API request (using `test-e2e.js` hitting port 5000) and the successful direct backend script runs.

Core Mismatch 1: CPU Saturation & Keep-Alives
Running bulk imports with 100+ records requires executing 100 concurrent `bcrypt.hash` operations. Each hash takes 80–100ms. Since Node.js runs asynchronous operations on a pool (libuv) with a default size of 4 threads, launching 100 concurrent bcrypt calls saturates the thread pool for 4–7 seconds. During this saturation, the Node process is unable to handle intermediate proxy keep-alives or DB queries, triggering connection drops or timeout resets at the proxy/Nginx layer, resulting in HTTP 502/500 errors.

Core Mismatch 2: Skipped-Row Foreign Key Constraint Failures
If the CSV contains duplicate records (e.g. from previous tests), `tx.user.createMany({ skipDuplicates: true })` silently skips them. However, `tx.studentProfile.createMany` still attempts to insert student profiles referencing the generated user IDs. Because the parent user row was skipped, this causes a database foreign key violation (`P2003`), crashing the transaction and returning HTTP 500.

First Point of Divergence & Failure Mechanisms

Failure Mechanism A: Libuv Thread Pool Saturation (Performance Bottleneck)

When processing 100 concurrent bcrypt hash operations, the Node process uses libuv threads to offload the cryptography computations. Because each hash is CPU-intensive, the thread pool is fully saturated for 4–7 seconds. Intermediate network proxies (like Nginx) or client connections may interpret this silence as an unhandled socket drop, closing the socket and returning a 500/502.

Failure Mechanism B: Prisma Foreign Key Constraint (Data Layer Mismatch)

If any valid row contains a user that has already been created in a previous run, `tx.user.createMany` skips the insert. However, the transaction block still attempts to insert a record in the `student_profiles` table referencing the generated user ID. Because the parent user creation was skipped, this results in a foreign key constraint violation (`student_profiles_user_id_fkey`) which aborts the entire transaction block, raising a Prisma exception caught as an HTTP 500 error: 

PrismaClientKnownRequestError: Foreign key constraint failed on the field: `student_profiles_user_id_fkey` (P2003)

Corrective Fixes

  1. Bcrypt Thread Pool Optimization:
    Increase the default libuv thread pool size by configuring the environment variable before bootstrapping the node process: 
    process.env.UV_THREADPOOL_SIZE = 128; // Add this to the very top of src/server.ts
  2. Prisma Duplicate User & Profile Safety:
    Filter the student profile inserts to ensure they are only inserted for users that were not skipped, or resolve user creations individually inside the transaction loop: 
    // Recommended update in UserProcessor.ts to prevent foreign key violations:
for (const row of batch) {
  const user = await tx.user.findUnique({ where: { email: row.email } });
  if (!user) {
    const created = await tx.user.create({ data: usersDataRow });
    if (row.admission_number || row.mobile_number) {
      await tx.studentProfile.create({ data: { user_id: created.id, ... } });
    }
  }
}
